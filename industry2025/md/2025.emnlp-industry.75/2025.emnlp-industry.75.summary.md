# Summary

## TL;DR  
The paper presents a test‑time reinforcement learning (RL) agent that improves schema‑mapping accuracy without any labeled data or model weight updates. By iteratively identifying ambiguous mappings, issuing targeted web‑search queries for evidence, and using a confidence‑based reward signal, the agent raises mapping accuracy from 72.73 % (baseline) to 93.94 % over 100 iterations on Microsoft Defender for Endpoint logs, while cutting low‑confidence mappings needing expert review by 85 %. The approach offers a transparent, evidence‑driven method suitable for industrial log‑integration pipelines.

## Problem & Motivation  
Enterprise platforms must ingest terabytes of heterogeneous logs from third‑party vendors (firewalls, endpoints, cloud apps, etc.). Vendor documentation is often missing, mismatched, or poorly formatted, making schema mapping—critical for downstream correlation and threat detection—highly error‑prone. Existing LLM‑based methods (prompting, fine‑tuning, retrieval‑augmented generation) struggle when no ground‑truth labels exist and documentation is incomplete. The paper targets this real‑world challenge: mapping poorly documented vendor schemas to a stable target schema without costly model updates or labeled data.

## Approach  
The core idea is a **test‑time RL agent** that self‑improves by **verbal memory updates** rather than weight fine‑tuning.  
1. **State**: current mapping hypothesis \(M_t\) and accumulated evidence \(E_t\).  
2. **Action**: select an ambiguous source field, formulate a targeted search query, and collect external evidence.  
3. **Reward**: change in **confidence** \(r_t = C_{t+1} - C_t\), where confidence is the consistency of predictions across \(n=3\) prompt variants.  
4. **Policy**: an LLM (GPT‑40) generates mappings conditioned on the current evidence context.  
5. **Conflict detection**: disagreements among the 3 predictions for a field trigger evidence collection.  
6. **Evidence handling**: only evidence that increases confidence is retained; otherwise discarded.  
The process is formalized in Algorithm 1, iterating up to \(\alpha=100\) times.

## Experimental Setup  
- **Datasets**:  
  - **Source**: Microsoft Defender for Endpoint schema (195 fields).  
  - **Target**: Common Schema (137 fields).  
  - **Ground truth**: 66 manually verified field mappings (curated by domain, threat‑intelligence, and product experts).  
- **Model**: GPT‑40 (GPT‑4o).  
- **Metrics**:  
  - **Accuracy**: % of correctly predicted mappings among the 66 verified pairs.  
  - **Confidence**: consistency of predictions across 3 attempts (modified frequency‑based, empty predictions weighted 0.5).  
- **Baselines**:  
  1. **Baseline 1** – GPT‑40 with single‑shot prompt (field name & value) → 56.36 % accuracy.  
  2. **Baseline 2** – GPT‑40 with single‑shot prompt + field descriptions, data types, and sample data → 72.73 % accuracy.  
- **Iterations**: 100 test‑time RL iterations.  
- **Evidence**: 81 evidence tuples collected across 100 iterations.  
- **Conflict count**: decreased from 26 to 4 (85 % reduction).

## Results  
- **Accuracy trajectory**:  
  - Iteration 1: +21.21 % (from 72.73 % to 93.94 %).  
  - Iteration 5: +9.60 %.  
  - Iteration 6: +10.71 %.  
  - Final accuracy after 100 iterations: **93.94 %**.  
- **Confidence**: steadily increased, approaching 1.0, indicating growing certainty.  
- **Conflict reduction**: from 26 to 4 fields (85 % drop).  
- **Expert review**: low‑confidence mappings requiring review dropped by **85 %**.  
- **No ground‑truth needed**: confidence scores guided RL, enabling deployment without labeled data.
