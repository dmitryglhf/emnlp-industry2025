# Summary

## TL;DR  
TOBUGraph replaces traditional Retrieval‑Augmented Generation (RAG) by automatically building a knowledge graph from unstructured user memories and retrieving via graph traversal. In the TOBU personal‑memory app, TOBUGraph achieves **93.75 % precision, 91.96 % recall, and 92.84 % F1**—a ~7 % lift over the best RAG baseline—and is preferred by users 75 % of the time in pairwise evaluations.

## Problem & Motivation  
RAG relies on vector‑space similarity between a query and individual text chunks, which ignores deeper semantic relationships across chunks, is highly sensitive to chunking strategy, and can hallucinate when relevant chunks are missing. In a commercial setting such as the TOBU app, these limitations manifest as low recall, inconsistent retrieval, and poor user experience. The authors aim to overcome these issues by constructing a dynamic knowledge graph that captures richer relationships among memories.

## Approach  
1. **Data Capture** – A multimodal LLM extracts object, emotion, scene, and geolocation details from user‑uploaded images/videos, producing an initial summary.  
2. **Graph Construction** –  
   * *Individual Memory Structuring*: Each memory becomes a *memory node* connected to *semantic nodes* (date, location, summary, etc.).  
   * *Cross‑Memory Relationship Discovery*: An LLM analyzes all memory nodes to identify shared themes (e.g., hobbies, locations). For each theme a *relationship node* is created and linked to all relevant memory nodes, forming the Relational Memory Graph (RMG).  
3. **Retrieval** – For a query *q*, an LLM first selects relevant relationship nodes *R′* (`f(q,R) → R′`). Then memory nodes connected to *R′* are retrieved (`g(R′, G′) → M′`). Finally, the conversational AI assistant generates a response from the retrieved memories and their semantic content (`h(q, M′, S′) → r`).  
4. **Implementation** – The system uses LangChain and ChromaDB for baseline RAG comparisons; TOBUGraph itself stores the graph in memory‑specific data structures and performs traversal at query time.

## Experimental Setup  
| Item | Details |
|-------|----------|
| **Dataset** | 20 highly active TOBU users, 80 unique memory‑retrieval requests (real user data, anonymized). |
| **Baselines** | Three RAG variants (LangChain + ChromaDB):<br>• **RAGv1** – fixed‑size chunks with overlap from AI‑generated summaries.<br>• **RAGv2** – each conversation with the AI assistant is a single chunk.<br>• **RAGv3** – each summary is a single chunk. |
| **Metrics** | Precision, Recall, F1‑score (95 % confidence intervals). |
| **User Study** | 480 evaluators, 4,800 pairwise comparisons (each approach evaluated 1,200 times). Tool: SLAM (Irugalbandara et al., 2024). |
| **Evaluation Levels** | Four request categories (Level 1–4) based on retrieval complexity (Table 3). |

## Results  
| Metric | RAGv1 | RAGv2 | RAGv3 | TOBUGraph |
|--------|--------|--------|--------|-----------|
| Precision (%) | 85.92 | 86.30 | 89.23 | **93.75** |
| Recall (%) | 66.40 | 79.60 | 82.26 | **91.96** |
| F1‑Score (%) | 74.53 | 82.88 | 85.56 | **92.84** |

*User Preference* – In pairwise comparisons, TOBUGraph responses were chosen 75 % of the time on average (Figure 2). Preference remained high across all request levels (Figure 3), whereas RAG baselines dropped as complexity increased.

*Qualitative Findings* – TOBUGraph consistently retrieved all relevant memories via relationship nodes (e.g., “California” node), whereas RAG missed memories beyond the top‑k chunk limit (I1 in Table 4). Users cited smoother flow and richer detail in TOBUGraph responses.

## Industry Impact  
TOBUGraph is deployed in the production TOBU app for personal memory organization and retrieval. It eliminates chunking configuration, reduces hallucinations, and improves retrieval accuracy, directly enhancing user experience in a commercial product. The authors plan to open‑source the dataset and experimental code for further study.
