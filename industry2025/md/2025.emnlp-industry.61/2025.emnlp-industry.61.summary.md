# Summary

## TL;DR
The paper presents a hybrid framework that injects LLM‑generated sentence embeddings into a foundation model for transaction understanding. By pre‑computing semantic embeddings for merchant name, MCC, and location fields and initializing the model’s embedding layers with them, the authors achieve significant performance gains on a billion‑record transaction dataset while keeping inference latency low enough for real‑time deployment.

## Problem & Motivation
Payment networks generate massive transactional data that encode consumer and merchant behavior. Existing foundation models for transaction analysis process tabular data sequentially but encode categorical merchant fields as discrete indices, losing rich semantic information (e.g., “Costco” is mapped to a single token). This semantic impoverishment limits the model’s ability to capture business‑type relationships and cold‑start behavior. LLMs can understand textual merchant information but are too slow for real‑time financial inference. The paper therefore seeks a production‑ready method to bring LLM semantic understanding into transaction models.

## Approach
The core idea is a **semantic initialization pipeline**:

1. **Data Collection & Fusion** – Gather raw categorical merchant data (merchant name, MCC, location) and enrich it with external sources (official MCC docs, business type annotations, demographic indicators, etc.).
2. **Prompt Generation with One‑Word Limitation** – Convert enriched data into structured prompts that explicitly ask the LLM to produce a *single‑word* representation, ensuring consistent, focused outputs across different LLMs.
3. **LLM Processing & Embedding Extraction** – Run the prompts through four open‑source LLMs (Llama2‑7b, Llama2‑13b, Llama3‑8b, Mistral‑7b) and extract the final hidden layer’s last non‑padding token as the sentence embedding.
4. **Foundation Model Initialization** – Use these embeddings to initialize the categorical field embedding layers of the existing tabular foundation model (Yeh et al. 2025). The rest of the architecture and training procedure remain unchanged.
5. **Multi‑task Learning** – Jointly optimize for five downstream tasks (next amount, next MCC, next location, next merchant, transaction‑metric assessment) to learn versatile representations.

Key design choices:  
- **One‑word limitation** to avoid verbose or inconsistent LLM outputs.  
- **Multi‑source data fusion** to capture richer context for each categorical field.  
- **Offline pre‑computation** of embeddings to keep inference latency low.

## Experimental Setup
- **Dataset**: 1 billion transaction records from Jan 2022–Dec 2023.  
  - 20 months for training, 21st month for validation, final 3 months for testing.  
- **Baselines**: Vanilla foundation model (ID‑based categorical embeddings).  
- **LLMs**: Llama2‑7b, Llama2‑13b, Llama3‑8b, Mistral‑7b.  
- **Embedding Initialization Strategies**: MCC, Merchant, MCC + Merchant, State City, All Fields.  
- **Tasks**:  
  1. Next Amount Prediction (regression) – MAE, sMAPE.  
  2. Next MCC Prediction (multi‑class) – Accuracy, F1.  
  3. Next Location Prediction (city) – Accuracy, F1.  
  4. Next Merchant Prediction (multi‑class) – Accuracy, F1.  
  5. Transaction Metrics Assessment – in‑house metric, reported as Relative Improvement (RI).  
- **Metrics**: MAE, sMAPE, Accuracy, F1, RI (formula given in the paper).
