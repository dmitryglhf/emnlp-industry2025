# Summary

## TL;DR  
The paper presents a unified evaluation pipeline for assessing data selection and synthesis methods in mathematical reasoning, showing that *better‑structured* and *model‑distilled* data consistently outperform sheer volume increases. In an industrial setting, the authors demonstrate that integrating curated datasets (e.g., OpenR1‑Math‑220K, Math‑Cosmo, NaturalReasoning‑QwQ) yields measurable gains (up to +8.96 % on MATH) while keeping deployment complexity low.

## Problem & Motivation  
High‑quality training data is essential for large language models (LLMs), yet most synthetic data construction methods are evaluated only in academic settings. In industrial pipelines, models must handle multiple tasks simultaneously, and naïvely adding more data can introduce noise and degrade performance. The paper investigates whether “more data” or “better data” is more valuable for mathematical reasoning in real‑world deployments.

## Approach  
The authors adopt a **unified evaluation pipeline** that mirrors both pre‑training and supervised fine‑tuning (SFT) stages.  
- **Base model**: DeepSeek‑V2‑Lite or Qwen2.5‑3B.  
- **Baseline**: the base model trained on a large validated dataset containing both code and mathematical data.  
- **Data integration**: new datasets are mixed with the validated set at a 0.2 weight (20 % new data, 80 % baseline).  
- **Annealing**: training follows the annealing schedule from Grattafiori et al. and Gu et al.  
- **Benchmarks**: four task types—common knowledge, logical reasoning, mathematical reasoning, coding—evaluated with accuracy (or Pass@1 for coding).  
- **Data selection**: evaluate open‑source datasets (IndustryCorpus2, opc‑fineweb‑math‑corpus, LIMO, s1K, OpenR1‑Math‑220K).  
- **Data synthesis**: pre‑training refinement (Cosmopedia, Math‑Cosmo) and SFT generation (S1, SynthLLM, Dolphin‑R1, NaturalReasoning‑QwQ).  
- **Model‑distilled data**: use Qwen2.5‑72B, QwQ‑32B, Mixtral‑8x7B‑Instruct‑v0.1, Math‑BERT, FAISS, MinHash deduplication.

## Experimental Setup  
| Component | Details |
|-----------|----------|
| **Base model** | DeepSeek‑V2‑Lite or Qwen2.5‑3B |
| **Baseline training** | Large validated dataset (code + math) |
| **New dataset weight** | 0.2 (20 % new data) |
| **Annealing schedule** | Grattafiori et al., Gu et al. |
| **Benchmarks** | MMLU, MMLU‑Pro, CMMLU, GPQA‑Diamond (knowledge); HellaSwag, BBH, DROP (reasoning); MATH, GSM8K, MathBench‑a, MathBench‑t (math); OpenAI‑HumanEval, Sanitized‑MBPP (coding) |
| **Metrics** | Accuracy (knowledge, reasoning, math); Pass@1 (coding) |
| **Datasets evaluated** | IndustryCorpus2, opc‑fineweb‑math‑corpus, LIMO, s1K, OpenR1‑Math‑220K, Math‑Cosmo, NaturalReasoning‑QwQ, etc. |
| **Data sizes** | OpenR1‑Math‑220K: 220 k samples; Math‑Cosmo: 760 M tokens; Math‑Retrieval: 75 k examples (82 M tokens) |

## Results  
| Benchmark | Score | Δ |
|-----------|-------|----|
| MMLU | 64.95 | –0.46 |
| MMLU‑Pro | 14.58 | +1.39 |
| CMMLU | 78.06 | +0.15 |
| GPQA‑Diamond | 32.32 | +0.5 |
| HellaSwag | 81.12 | –0.67 |
| BBH | 63.87 | +1.16 |
| DROP | 63.48 | +0.64 |
| MATH | 49.68 | +1.92 |
| GSM8K | 81.50 | –0.99 |
| MathBench‑a | 38.07 | +1.47 |
| MathBench‑t | 63.84 | +0.07 |
| OpenAI‑HumanEval | 57.93 | –0.61 |
| Sanitized‑MBPP | 53.7 | –0.77 |

- **OpenR1‑Math‑220K**: +8.96 % accuracy on MATH.  
- **Math‑Cosmo**: +1.72 % accuracy on MATH.  
- **NaturalReasoning‑QwQ**: +1.92 % accuracy on MATH.  

Other synthesis methods (weakness‑augmented, retrieval‑augmented) showed negligible gains, confirming that data quality, not quantity, drives performance.
