# Summary

## TL;DR  
RLTR introduces a single‑objective reinforcement learning framework that focuses exclusively on the planning module of LLM agents. By replacing the final‑answer reward with a tool‑use completeness signal, RLTR decouples planning from summarization, alleviates credit‑assignment issues, and yields an 8–12 % boost in planning accuracy and a 5–6 % lift in overall response quality on industry‑level Chinese datasets.

## Problem & Motivation  
- **Core challenge**: In real‑world agent pipelines, the planning stage (tool invocation) is critical, yet existing end‑to‑end RL methods jointly optimize planning and summarization using the final answer as reward.  
- **Data scarcity**: Most industrial data lack verifiable ground truth, making answer‑based rewards unreliable and prone to reward hacking.  
- **Credit assignment**: Planning and summarization gradients often conflict; a wrong final answer can penalize otherwise correct tool calls.  
- **Industry context**: The work originates from WeChat, Tencent, targeting Chinese LLM agents that must handle search and code tools in production settings.

## Approach  
1. **Cold‑start Planner** – Knowledge distillation from a strong teacher LLM followed by rejection sampling to select the best‑of‑n action trajectories.  
2. **Tool‑Use Completeness Reward** – A verification LLM (Comp. Checker) evaluates whether the planner’s action sequence is complete (`γ(s) ∈ {0,1}`) and aggregates over N samples:  
   \[
   R_{\text{comp}}=\frac{1}{N}\sum_{i=1}^{N}\gamma_i(\tau)
   \]  
3. **Multi‑turn RL** – The total reward combines completeness with rule‑based penalties for repetition (`R_{\text{repeat}}`) and incorrect tool usage (`R_{\text{error}}`):  
   \[
   R_{\text{total}}=
   \begin{cases}
   -1 & \text{if format invalid}\\
   R_{\text{comp}}+R_{\text{rule}} & \text{otherwise}
   \end{cases}
   \]  
   The planner is optimized with a KL‑regularized policy gradient objective (Eq. 4).  
4. **Modular Agent** – The trained planner (`π_p*`) generates a tool‑call trajectory; any LLM can serve as the summarizer (`π_s`) to produce the final answer.

## Experimental Setup  
| Component | Details |
|-----------|----------|
| **Datasets** | • Industry: ~4 k train, 0.5 k test (normal/hard split).<br>• Open‑source: ChineseSimpleQA (filtered to 855 tool‑required samples). |
| **Baselines** | • Qwen3‑235B‑A22B, DeepSeek‑R1 (direct tool use).<br>• End‑to‑end SFT (E2E SFT).<br>• End‑to‑end RL (E2E RL). |
| **Metrics** | • Planning: Tool‑use completeness (Com.).<br>• Final response: Match (ground truth), Hel. (helpfulness), Rel. (relevance). |
| **Training** | Offline evaluation; RL reward computed by Qwen3‑235B‑A22B (answer‑based) vs. RLTR (completeness‑based). |
| **RL Algorithms** | PPO, GRPO, REINFORCE++ (all within RLTR). |
