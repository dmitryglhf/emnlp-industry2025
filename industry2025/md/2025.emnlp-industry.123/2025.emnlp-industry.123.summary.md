# Summary

## TL;DR  
The paper introduces two new Spider‑derived benchmarks that require both SQL and API calls, and proposes a declarative “siwarex” framework that treats APIs as user‑defined functions (UDFs) in SQL.  Using this approach, the authors show that it outperforms state‑of‑the‑art imperative code‑generation and ReAct‑based agentic systems on the new benchmarks.

## Problem & Motivation  
Industrial users often ask natural‑language questions that must be answered by pulling data from a mix of structured tables, spreadsheets, and external APIs.  Existing Text‑to‑SQL benchmarks (Spider, Spider‑2, BIRD) and tool‑calling benchmarks (BFCL, Toolformer, ToolLLM, NESTFUL, XLAM) evaluate only one data source type.  Consequently, systems that can translate a question into a single SQL query or a single API call cannot be evaluated on realistic heterogeneous workloads.  The authors argue that agentic architectures such as ReAct are brittle, expensive, and hard to scale, because they conflate intent representation with execution planning.

## Approach  
The core idea is to keep the user’s intent declarative (SQL) and to place APIs on the same footing as database tables by wrapping them in UDFs.  The siwarex framework requires two schemas:  
1. **Abstract Schema** – an ER diagram that is agnostic to whether a source is a table or an API.  
2. **API Mapping Schema** – contains URL, HTTP method, and input/output details for each API.  

From the Abstract Schema a relational schema is generated automatically.  Tables that correspond to APIs are *virtual tables* whose rows are produced by a UDF that calls the API.  Two variants are implemented:  
* **Declarative** – uses a rule‑based Query Rewriter to replace virtual tables with their UDFs in the generated SQL.  
* **Declarative2** – for scalar APIs, the UDF is called directly in the SQL expression (e.g., `SELECT LENGTH(name) FROM people`).  

All three query strategies (Declarative, Declarative2, Imperative, Agentic) use the same LLM, Mistral‑Large.  The Imperative approach generates a Python program that orchestrates database and API calls.  The Agentic approach is a ReAct agent that can call two tools:  
* **SQLDatabaseToolkit** (LangChain) for SQL queries, and  
* **xLAM** (the top‑ranked Berkeley API Leaderboard tool as of mid‑2025) for API calls.  

The agent receives the same metadata (table schemas, API descriptions) and produces a chain of thought that lists the required SQL and API calls.

## Experimental Setup  
- **Benchmarks**:  
  * **Benchmark I** – 948 Spider questions with ground‑truth SQLs; a subset of Spider tables are replaced by equivalent APIs (e.g., `/museum`).  
  * **Benchmark II** – 2 338 question/answer pairs that require 1–3 scalar APIs (lexical, numeric, geospatial).  The scalar APIs include 16 functions such as `count_syllables`, `is_prime`, and Google Geospatial wrappers (`get_country_of_place`, `distance_between`).  
- **Baselines**: Imperative code‑generation and Agentic ReAct.  
- **LLM**: Mistral‑Large for all three strategies.  
- **Tools**: SQLDatabaseToolkit (LangChain) and xLAM.
