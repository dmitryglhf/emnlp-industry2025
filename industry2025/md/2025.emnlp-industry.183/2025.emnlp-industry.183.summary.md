# Summary

## TL;DR
The paper proposes a three‑stage framework that first filters a tabular dataset for inlier rows, then uses the LLM **Gemma‑3‑12B** to generate semantically rich data‑quality (DQ) rules, and finally translates those rules into executable Python code with **Qwen‑2.5‑Coder (32 B)**.  The approach leverages retrieval‑augmented generation, domain‑aware few‑shot prompts, and guardrails (conflict‑resolution and rubric‑based evaluation) to produce reliable, high‑value DQ rules.  Experiments on ten benchmark datasets (Adult, Beers, Bikes, Breast Cancer, HAR, Movies, Nasa, Rayyan, Soil Moisture, Tax) with synthetic noise (10 %, 20 %, 30 %) show that the full pipeline outperforms a suite of statistical baselines (ED2, FA‑HES, KATARA, OUTLIER IQR, OUTLIER IF, OUTLIER SD, MAX‑ENTROPY, MIN‑K).

## Problem & Motivation
Reliable data quality is essential for downstream analytics, yet rule‑based validation is often inefficient, requires human intervention, and incurs high computational costs.  Statistical profiling (distribution shifts, outliers, functional dependency violations) captures structural anomalies but lacks semantic understanding, missing context‑dependent errors that rely on external knowledge.  Existing LLM‑based detectors (fine‑tuned models, prompt‑driven detectors, in‑context repair) are computationally expensive or require dataset‑specific training.  The paper addresses these gaps by combining statistical filtering with LLM‑driven rule generation and code synthesis, aiming for scalable, semantically aware DQ assessment.

## Approach
The framework consists of three stages:

1. **Inlier–Outlier Detection** – Uses the **Sparx** algorithm (distributed on Apache Spark) to perform multivariate outlier detection at the row level.  String columns are embedded with **BERT** and evaluated with a distance‑based test; numeric and categorical columns use type‑specific detectors.  A row is marked an outlier if the number of its outlier cells exceeds a threshold.  Only the inlier subset proceeds to the next stages.

2. **Generation & Enrichment of DQ Rules** – The LLM **Gemma‑3‑12B** is prompted to produce a *Rule Card* for each DQ dimension (Accuracy, Conformity, Completeness, Consistency).  The prompt includes a schema description, domain‑aware few‑shot examples (retrieved via **all‑MiniLM‑L6‑v2**), and a knowledge section.  An enrichment phase adds specification, reference tables, and concise pseudocode.  Guardrails include:
   * **Conflict‑Resolution Filter** – Detects mutually exclusive or contradictory rules using a dedicated Gemma prompt.
   * **Rubric‑Based Evaluation** – An eight‑point rubric (Duplicate, Redundant, Trivial, Risk‑false‑positive, Miscategorized, Ambiguous, Hallucinated‑overly‑specific, High‑value, Correct, Correct‑fixable) classifies rules; only *High‑value*, *Correct*, or *Correct‑fixable* proceed.

3. **Python Code Generation** – Each validated rule card is translated into a `check(df)` function by **Qwen‑2.5‑Coder (32 B)**.  The prompt supplies table schema, column profiles, the enriched rule card, sample rows, and domain‑specific code snippets.  Generated code is executed on a few rows; any exception is fed back to the LLM for repair.

The end‑to‑end workflow aggregates error masks from all generated functions and produces a *Quality Assessment Report* listing each flagged cell, the triggering rule, and the responsible code snippet.

## Experimental Setup
- **Datasets**: Ten well‑known tabular datasets – *Adult*, *Beers*, *Bikes*, *Breast Cancer*, *HAR*, *Movies*, *Nasa*, *Rayyan*, *Soil Moisture*, *Tax*.
- **Noise Injection**: Synthetic errors are introduced using the standardized **REIN** corruption model at 10 %, 20 %, 30 % of cells.  Error types include keyboard typos, explicit/implicit missing values, cell swaps, and Gaussian noise.
- **Baselines**: ED2, FA‑HES, KATARA, OUTLIER IQR, OUTLIER IF, OUTLIER SD, MAX‑ENTROPY, MIN‑K.

## Results
- The full pipeline achieves higher F<sup>1</sup> scores than all baselines across all noise levels.
- Ablation without the inlier module shows a significant drop in F<sup>1</sup> scores, underscoring the importance of filtering noisy rows before rule generation.
- The approach consistently outperforms statistical baselines, demonstrating the benefit of LLM‑driven rule generation and code synthesis.
