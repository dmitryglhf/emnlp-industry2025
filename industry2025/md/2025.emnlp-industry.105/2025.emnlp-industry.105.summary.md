# Summary

## TL;DR  
FlexDoc is a scalable synthetic data generation framework that uses stochastic schemas and parameterized sampling to produce multilingual, semi‑structured documents with accurate annotations. On Key Information Extraction (KIE) tasks, FlexDoc‑generated data boosts F1 scores by up to **11 %** and cuts annotation effort by over **90 %** compared to hard‑template methods, and is already in active deployment for enterprise document‑understanding pipelines.

## Problem & Motivation  
Enterprise document‑understanding systems (e.g., invoice processing, claims automation) require large, diverse, and well‑annotated datasets. Collecting such data is prohibitively expensive because of privacy constraints, legal restrictions, and the manual effort needed to annotate layout, text, and visual elements. Existing hard‑template generators are limited: they inherit the layout of a single real document, cannot model positional variation, and scale linearly with annotation time. FlexDoc addresses these gaps by enabling controlled, high‑diversity synthetic generation without the need for real‑world data.

## Approach  
FlexDoc’s pipeline consists of three core components:

1. **Stochastic Schema** – a JSON‑based specification where each entity group and entity is defined as a random variable (e.g., number of rows, presence probability, font, color).  
2. **Parameterized Sampling Algorithm** – samples all stochastic attributes to produce a *document permutation*; fake values are generated with the Python `faker` library (or custom generators) and translated via a machine‑translation engine for multilingual support.  
3. **Dynamic Virtual Grid Algorithm** – treats each section as a virtual grid; entity groups are placed into cells with dynamic row/column sizing to avoid overlap, preserve whitespace, and maintain visual structure.  
The final **Document Rendering** step draws the sampled layout onto a blank canvas using Pillow, recording bounding boxes for key‑value labels, table boundaries, and other KIE annotations.

Key design choices:  
- Use of `faker` for locale‑aware fake data.  
- Translation of header strings to target language via a translation engine.  
- Fixed annotation effort (150 min) regardless of dataset size, versus linear growth for hard‑template methods.

## Results  
### Incremental Synthetic Augmentation  
| Train Dataset | LayoutLM | Phi‑4 |
|---------------|----------|-------|
| Zero Shot | NA | 31.2 |
| Synthetic 5k Only | 54.1 | 53.3 |
| DocILE Only (Baseline) | 72.7 | 74.6 |
| DocILE + 1k Synthetic | 76.3 | 79.8 |
| DocILE + 2k Synthetic | 78.8 | 82.4 |
| DocILE + 3k Synthetic | 81.8 | 83.1 |
| DocILE + 4k Synthetic | 82.7 | 85.3 |
| DocILE + 5k Synthetic | **82.9** | **85.6** |

| Train Dataset | LayoutLM | Phi‑4 |
|---------------|----------|-------|
| Zero Shot | NA | 23.1 |
| Synthetic 5k Only | 47.3 | 44.4 |
| IDSEM Only (Baseline) | 65.5 | 71.2 |
| IDSEM + 1k Synthetic | 67.3 | 74.5 |
| IDSEM + 2k Synthetic | 69.1 | 76.0 |
| IDSEM + 3k Synthetic | 73.3 | 77.3 |
| IDSEM + 4k Synthetic | 74.2 | 78.6 |
| IDSEM + 5k Synthetic | **75.6** | **79.5** |

*Absolute improvements*: LayoutLM gains **+10.2 %** on DocILE and **+10.1 %** on IDSEM; Phi‑4 gains **+11 %** and **+8.3 %** respectively.

### Ablation – Dynamic Virtual Grid  
With the grid algorithm disabled, LayoutLM’s F1 drops from **82.9 %** to **75.4 %**; Phi‑4 drops from **85.9 %** to **80.6 %**.

### Hard‑Template vs. FlexDoc  
| Approach | LayoutLM | Phi‑4 | Annotation Effort (mins) |
|----------|----------|-------|--------------------------|
| Hard Template | 69.8 | 72.2 | 1500 |
| FlexDoc | 67.4 | 69.8 | 150 |

FlexDoc achieves comparable performance while reducing annotation effort by **≈90 %**.
