# Summary

## TL;DR  
TTD‑SQL introduces a lightweight, model‑agnostic decoding framework that pre‑computes token‑level decision trees for SQL keywords, table names, and column identifiers. By deterministically “auto‑filling” uniquely determined tokens, it cuts inference latency by up to 19.96 % tokens/s and raises executable‑SQL rates by +17.7 % across five large language models, making real‑time, schema‑aware SQL generation feasible for enterprise analytics.

## Problem & Motivation  
- **Inference latency**: Autoregressive LLMs require a full forward pass per token; on BIRD (Minidev) Qwen2.5‑Coder‑32B takes 14.3 s/query and Llama‑70B 22.86 s/query.  
- **Schema hallucinations**: Models often emit invalid column references (e.g., `users.UserId` instead of `users.Id`).  
- **Real‑world impact**: In finance, healthcare, and enterprise analytics, both latency and schema validity are critical for user trust and regulatory compliance. Existing grammar‑constrained or schema‑linking methods either add parser overhead or require fine‑tuning, limiting practical deployment.

## Approach  
- **Token‑chain transition tree**: Build a directed tree over the LLM’s vocabulary where nodes represent prefixes of schema elements or SQL keywords.  
- **Deterministic auto‑fill**: If a token chain has a single valid continuation, the decoder selects it without an LLM forward pass.  
- **Hybrid decoding**: For ambiguous contexts, the LLM’s standard sampling (greedy or top‑k) is used.  
- **Tree invocation rules**:  
  - *Keyword trees* auto‑complete SQL keywords (e.g., `ORDER → BY`).  
  - *Schema table trees* trigger after `FROM`, `JOIN`, or `ON`.  
  - *Schema column trees* activate after a dot following a table name.  
- **Offline construction**:  
  1. Augment schema strings (case variants, quoted forms).  
  2. Tokenize with the target LLM’s tokenizer.  
  3. Assemble the prefix tree from token chains.  
- **Design choices**:  
  - **Model agnosticism**: Works with CodeLlama, Phi‑4, Qwen2.5, Granite, and Llama‑70B.  
  - **Tokenization sensitivity**: Finer tokenization (e.g., CodeLlama splits `cust_id` into `[cust, _, id]`) yields deeper decision paths and higher auto‑fill rates.  
  - **No model modification**: The framework is a post‑processing layer, avoiding white‑box changes or beam‑search overhead.

## Experimental Setup  
| Dataset | Size | Description |
|---------|-------|-------------|
| **Spider** | 992 dev questions from 19 DBs | Public benchmark |
| **BIRD (Minidev)** | 500 pairs from 11 DBs | Lightweight dataset |

| Model | Size | URL |
|-------|------|-----|
| CodeLlama 7B‑Instruct | 7B | https://huggingface.co/meta-llama/CodeLlama-7b-Instruct-hf |
| Microsoft Phi‑4 | 13B | https://huggingface.co/microsoft/phi-4 |
| Qwen2.5‑Coder‑32B‑Instruct | 32B | https://huggingface.co/Qwen/Qwen2.5-Coder-32B-Instruct |
| Granite‑34B‑Code‑Instruct‑8K | 34B | https://huggingface.co/ibm-granite/granite-34b-code-instruct-8k |
| Llama‑3.3‑70B‑Instruct | 70B | https://huggingface.co/meta-llama/Llama-3.3-70B-Instruct |

**Decoding strategies**:  
- **AR** – standard autoregressive decoding.  
- **GCD** – grammar‑constrained decoding (Ugare et al.).  
- **TTD** – proposed tree‑guided decoding.  
- **CoT** – chain‑of‑thought prompting (Wei et al.).  
- **CoD** – chain‑of‑draft prompting (Xu et al.).

**Evaluation metrics**:  
- **Execution Accuracy (EX)** – matches ground‑truth results (Spider framework).  
- **Token Rate (TR)** – tokens per second.  
- **Executable SQL %** – proportion of syntactically valid SQL.  
- **Auto‑fill Rate** – proportion of tokens deterministically predicted by TTD.

**Hardware**: 1–4 NVIDIA A100 40 GB GPUs.  
**Regime**: Zero‑shot, no post‑hoc validation or re‑ranking.

## Limitations  
- **Static schema requirement**: TTD relies on pre‑computed decision trees for fixed schemas; dynamic or evolving databases would need efficient incremental updates.  
- **Incremental update design**: The paper notes that designing algorithms for incremental tree updates remains future work.
