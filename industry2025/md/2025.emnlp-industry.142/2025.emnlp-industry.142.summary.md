# Summary

## TL;DR  
The paper introduces a neurosymbolic framework that trains a fully interpretable, rule‑based RDF‑to‑text generator by orchestrating five large‑language‑model (LLM) agents.  The resulting system, written entirely in Python, requires no supervised data, runs in a few seconds on a single CPU, and achieves reference‑based scores comparable to fine‑tuned neural baselines while producing fewer hallucinations and more grammatical outputs.

## Problem & Motivation  
RDF‑to‑text generation is widely used in industry for converting knowledge‑graph triples into natural language.  Neural models dominate the field but suffer from lack of interpretability, hallucinations, and high inference cost.  Industrial deployments need controllable, fast, and transparent systems.  The paper addresses this gap by proposing a rule‑based approach that is trained without human references and can be inspected and edited by developers.

## Approach  
The training pipeline is driven by five LLM agents:

| Agent | Role |
|-------|-------|
| **Software Architect (SA)** | Designs the high‑level code structure (functions, signatures). |
| **Software Engineer (SE)** | Implements each function iteratively, receiving design, current code, and feedback. |
| **Evaluator** | Executes the generated program on unit tests, then asks an LLM whether the output correctly reflects the input. |
| **Test Engineer (TE)** | Generates unit tests by sampling predicates from the KG and producing 50 input‑output pairs per predicate, ensuring at least three tests per predicate. |
| **Code Analyst (CA)** | Analyzes failed tests, decides whether to redesign the system (feedback to SA) or to fix specific functions (feedback to SE). |

The process iterates until all unit tests pass or a maximum of 25 iterations (10 for GPT‑4.1) is reached.  The final output is a single Python file that can generalise to unseen data sharing the same schema.  The system is fully interpretable, controllable, and does not invoke any LLM at inference time.

## Experimental Setup  
| Component | Details |
|-----------|---------|
| **Datasets** | 5 total: WebNLG (standard test set + 3 GEM 2024 shared‑task datasets – GEM2 Counterfactual, GEM2 Fictional, GEM2 Factual) and OpenDialKG. |
| **Baselines** | *Fine‑tuned BART* (Lewis et al., 2020), *Prompted Llama 3.3 70B* (Touvron et al., 2024). |
| **LLMs used for training** | GPT‑4.1 (proprietary), Qwen 3 235B, Qwen 2.5 72B, Llama 3.3 70B (all open‑source). |
| **Training details** | 25 iterations (10 for GPT), repeated 3 times; best model chosen by number of passed unit tests.  Open‑source models were 4‑bit quantised via ollama.  Evaluator always used Llama 3.3 for consistency. |
| **Inference context** | Offline, single CPU.  Inference times measured on an Nvidia A40 GPU (for baselines) and an AMD EPYC 7313 CPU (for rule‑based). |
| **Metrics** | Reference‑based: BLEU, METEOR, BERTScore, BLEURT.  Reference‑less: LLM‑as‑a‑Judge (grammaticality, addition of unsupported facts, omission). |

## Results  
### Reference‑based metrics (WebNLG test set)  
| System | BLEU | METEOR | BERTScore | BLEURT |
|--------|------|--------|-----------|--------|
| Fine‑tuned BART | **0.4352** | 0.6791 | **0.9308** | 0.1275 |
| Prompted Llama 3.3 70B | 0.3616 | 0.6887 | 0.9255 | 0.1058 |
| Our rule‑based NLG (GPT‑4.1) | 0.3934 | **0.7069** | 0.9291 | **0.1841** |
| (Qwen 3 235B) | 0.3939 | 0.6759 | 0.9290 | 0.1767 |
| (Qwen 2.5 72B) | 0.3309 | 0.6531 | 0.9224 | 0.1193 |
| (Llama 3.3 70B) | 0.2858 | 0.6578 | 0.9179 | 0.0618 |

*Out‑of‑domain (OOD) scores* (BLEU, METEOR, BERTScore, BLEURT) are 0.3615, 0.7124, 0.9251, 0.1483 respectively for GPT‑4.1.  The rule‑based system matches or exceeds the neural baselines on METEOR and BLEURT, and is competitive on BLEU and BERTScore.

### Reference‑based metrics (OpenDialKG)  
| System | BLEU | METEOR | BERTScore | BLEURT |
|--------|------|--------|-----------|--------|
| Fine‑tuned BART | 0.9372 | 0.9849 | 0.9973 | 0.9340 |
| Prompted Llama 3.3 70B | 0.2040 | 0.6289 | 0.9104 | 0.1348 |
| Our rule‑based NLG (Qwen 3 235B) | **0.3472** | **0.6947** | **0.9239** | **0.1047** |
| (Qwen 2.5 72B) | 0.3413 | 0.7030 | 0.9265 | 0.2300 |
| (Llama 3.3 70B) | 0.3120 | 0.6517 | 0.9216 | 0.1882 |

Fine‑tuned BART still leads on OpenDialKG, but the rule‑based models outperform the prompted LLM on all metrics.

### Inference time  
| System | GPU | CPU |
|--------|-----|-----|
| Fine‑tuned BART | 249 s | 1910 s |
| Prompted Llama 3.3 70B | 6360 s | n/a |
| Our approach (GPT‑4.1) | – | **7 s** |

The rule‑based system achieves a 35× speed‑up on CPU versus BART on GPU and a 272× speed‑up when both run on CPU.

### Reference‑less evaluation (LLM‑as‑a‑Judge)  
| Dataset | Gram. | Add. | Om. |
|---------|-------|------|-----|
| WebNLG | 0.734 (GPT‑4.1) | 0.029 | 0.111 |
| GEM2 Counterfactual | 0.729 | 0.026 | 0.040 |
| GEM2 Fictional | 0.663 | 0.019 | 0.065 |
| GEM2 Factual | 0.635 | 0.049 | 0.126 |
| OpenDialKG | 0.730 | 0.034 | 0.108 |

The rule‑based system trained with GPT‑4.1 shows the highest grammaticality and the lowest hallucinations (Add.) compared to fine‑tuned BART and prompted Llama 3.3 70B.  The Llama 3.3 70B baseline has the best grammaticality but higher Add. scores.

### Ablation experiments  
| Ablation | Effect |
|----------|--------|
| **Abl. 1** – static system design (human) | Highly negative impact on all metrics. |
| **Abl. 2** – training examples from WebNLG training set instead of generated unit tests | Negative impact, confirming the necessity of generated unit tests. |
