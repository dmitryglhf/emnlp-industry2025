# Summary

## TL;DR  
QuackIR demonstrates that relational database management systems (DuckDB, SQLite, PostgreSQL) can serve as effective retrieval engines for large‑language‑model (LLM) retrieval‑augmented generation (RAG) workloads. By wrapping SQL‑based BM25, dense vector search, and hybrid RRF fusion inside the databases, QuackIR matches the performance of established IR toolkits (Anserini/Pyserini) while keeping the stack minimal.

## Problem & Motivation  
Enterprises increasingly rely on dedicated vector stores for RAG, adding extra complexity to their “AI stack.” Many already own relational databases (RDBMSes) as part of their data lakes. The paper argues that using these existing RDBMSes for retrieval could reduce operational overhead and simplify deployment.

## Approach  
QuackIR is a modular toolkit that mirrors the pipeline of Anserini/Pyserini (pre‑processing → indexing → retrieval → evaluation) but executes all stages inside the RDBMS.  
* **Sparse retrieval**: BM25 implemented via DuckDB’s `fts_index`, SQLite’s FTS5, and PostgreSQL’s full‑text search (simple and modified configurations).  
* **Dense retrieval**: Brute‑force “flat” search using DuckDB’s `array_cosine_similarity` or PostgreSQL’s `pgvector` cosine distance, with embeddings from BGE‑base‑env1.5.  
* **Hybrid retrieval**: Reciprocal Rank Fusion (RRF) combining sparse and dense results (k = 60).  
The toolkit disables built‑in tokenizers in DuckDB and uses Lucene’s Porter tokenizer via Pyserini for consistency.

## Experimental Setup  
* **Hardware**: Azure instance, 64 × AMD EPYC 9V74 80‑core, Ubuntu 24.04.2, 503 GB RAM.  
* **Dataset**: BEIR flat variant (concatenated fields).  
* **Baselines**: Anserini (Java) and Pyserini (Python).  
* **Metrics**: nDCG@10 for effectiveness; queries per second (QPS) for latency.  
* **Evaluation**: qrels integrated in QuackIR’s submodule.  
* **Scalability**: Datasets grouped by corpus size (|C|) and query count (|Q|). Larger datasets were omitted when latency became impractical.

**Observations**  
* DuckDB and SQLite achieve nDCG@10 within 0.01 of the Anserini baseline on most datasets.  
* PostgreSQL (simple) lags by 0.1–0.15 on average; the modified configuration improves results but still falls short of baseline.  
* SQLite is the fastest in latency, followed by DuckDB, then PostgreSQL.  
* Dense retrieval with DuckDB and PostgreSQL (pgvector) matches Anserini’s dense scores; SQLite does not support dense retrieval.  
* Hybrid RRF fusions (DuckDB, PostgreSQL) yield marginal gains over the best single modality.
