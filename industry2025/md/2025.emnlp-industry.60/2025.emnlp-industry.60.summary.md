# Summary

## TL;DR  
The paper introduces **Combo‑Eval**, a hybrid evaluation framework that fuses metric‑based scoring with LLM‑as‑a‑judge to assess natural‑language representations (NLRs) of SQL execution results. It also releases **NLR‑BIRD**, the first dedicated dataset for benchmarking NLR generation across 11 domains. Combo‑Eval achieves higher alignment with human judgments while cutting LLM calls by up to **61 %** in the ground‑truth scenario.

## Problem & Motivation  
Text‑to‑SQL systems are widely used in industry chat agents, but the downstream step of converting tabular query results into natural‑language summaries (NLRs) lacks robust evaluation. Existing metrics alone are unreliable, and LLM‑based judges are costly, leading to inaccurate or incomplete NLRs that can degrade user experience. The paper targets this gap, aiming to provide a reliable, efficient evaluation method suitable for real‑world deployments.

## Approach  
1. **Dataset** – Build **NLR‑BIRD** from the BIRD‑dev set (1534 NL questions) by executing the SQL queries and collecting human‑written NLRs for result sets with **rc + cc < 500** (4.3 % of samples excluded). The final set contains **1468** samples across 11 domains.  
2. **Evaluation Scenarios** –  
   * **GT**: compare model‑generated NLRs against the human‑written ground‑truth NLR.  
   * **UQDB**: compare against the concatenation of the user question and the raw result table (no ground‑truth NLR).  
3. **Evaluation Methods** –  
   * **Metrics‑as‑a‑Judge**: use ROUGE‑1 recall (chosen for its largest median gap between correct/incorrect NLRs), BERTScore, and cosine similarity; thresholds are tuned on a 25 % dev set.  
   * **LLM‑as‑a‑Judge**: prompt a judge LLM (default GPT‑4o) to label each NLR as correct/incorrect.  
   * **Combo‑Eval**: first filter with metric thresholds; only samples falling in the “pending” zone are sent to the LLM judge. This reduces the number of expensive LLM calls while preserving accuracy.

## Experimental Setup  
| Item | Details |
|------|---------|
| **Dataset** | NLR‑BIRD (1468 samples, 11 domains, rc + cc < 500) |
| **Generation Models** | Phind‑CodeLLama‑34B‑v2, Llama‑3.1‑70B‑Instruct, Llama‑3.1‑405B‑Instruct, GPT‑4o |
| **Judge LLMs** | GPT‑4o (default), Grok‑3‑mini, GPT‑4.1‑nano, GPT‑4o‑mini |
| **Metrics** | ROUGE‑1 recall, BERTScore, cosine similarity |
| **Evaluation Metrics** | Macro F1, per‑class F1, overall accuracy |
| **Dev/Test Split** | 25 % of GPT‑4o‑generated NLRs used as dev for threshold calibration; remaining samples used as test |
| **Human Evaluation** | 660 labeled NLRs (165 unique questions) used as ground truth for method comparison |

## Results  
| Model | Size | 3‑9 | 10‑49 | 50‑499 | Total |
|-------|------|------|--------|--------|-------|
| **Phind** | – | 0.80 | 0.60 | 0.52 | 0.65 |
| **Llama‑3.1‑70B** | 70 | 0.87 | 0.77 | 0.57 | 0.75 |
| **Llama‑3.1‑405B** | 405 | 0.89 | 0.83 | 0.59 | 0.78 |
| **GPT‑4o** | 4o | 0.91 | 0.62 | 0.30 | 0.63 |

**Combo‑Eval** reduced LLM calls by **61.43 %** (GT) and **24.48 %** (UQDB).  
Macro F1 scores (macro across classes) for the test set:

| Scenario | Metrics | LLM‑Judge | Combo‑Eval |
|----------|---------|-----------|------------|
| **GT** | 69.73 | 80.72 ± 0.8 | 80.88 ± 0.4 |
| **UQDB** | 68.06 | 75.82 ± 0.4 | 76.98 ± 0.4 |

Per‑class F1 values (class 0 / class 1) and standard deviations are reported in Table 2 of the paper. Combo‑Eval consistently outperforms both pure metric and pure LLM judges across all judge LLMs, with lower variance due to fewer LLM evaluations.

## Limitations  
* **Large Result Sets** – LLMs struggle with rc + cc ≥ 10, producing incomplete or hallucinated NLRs.  
* **Formatting Inconsistencies** – especially in GPT‑4o outputs, affecting readability.  
* **LLM‑Judge Accuracy Degrades** – for larger result sets, the judge LLM misclassifies more often.  
* **Dataset Scope** – only covers rc + cc < 500; real‑world systems may encounter larger tables.  
* **Combo‑Eval Still Limited** – while efficient, it cannot fully eliminate errors for very large tables.
