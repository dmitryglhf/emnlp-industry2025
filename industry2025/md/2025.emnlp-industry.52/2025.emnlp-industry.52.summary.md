# Summary

## TL;DR  
The paper introduces **MeBP (Memory‑Efficient Backpropagation)**, a backprop implementation that uses gradient checkpointing, lazy weight decompression, and memory‑mapped activations to fine‑tune LLMs on mobile devices. MeBP keeps the memory footprint below 1 GB for models from 0.5 B to 4 B parameters, converges faster than zeroth‑order (ZO) baselines, and achieves comparable or better performance while only marginally increasing per‑step compute time.

## Problem & Motivation  
Fine‑tuning large language models on mobile devices is impractical because backpropagation requires far more memory than inference. Existing on‑device methods such as **MeZO** (zeroth‑order optimization) reduce memory but need 10×–100× more optimization steps, still demanding several gigabytes of RAM (e.g., 6.5 GB for OPT‑1.3B). The authors aim to provide a first‑order method that fits within the DRAM limits of modern phones.

## Approach  
1. **LoRA fine‑tuning** – only linear layers are trainable, keeping trainable parameters < 1 % of the full model.  
2. **Base‑weight compression** – 4‑bit symmetric INT4 quantization on all frozen parameters (including embeddings).  
3. **Gradient checkpointing** – split the model into blocks; during forward pass only the necessary checkpoints are stored, and during backward pass they are recomputed on demand.  
4. **Lazy loading & decompression** – base weights are memory‑mapped and decompressed only when needed; activations are also memory‑mapped.  
5. **Runtime** – implemented in Swift for iOS; Algorithm 1 describes the forward–backward loop that loads checkpoints, recomputes gradients, and keeps only LoRA weights and gradients in RAM.

## Experimental Setup  
- **Baselines**: MeZO (zeroth‑order) and full‑parameter fine‑tuning (FFT).  
- **Datasets**:  
  - Language modeling: **WikiText‑2** (2,048 samples, seq. length 256).  
  - Function calling: **ToolACE** (2,672 samples, seq. length 512) for training; evaluation on **Berkeley Function Calling (BFCL) v4 live split**.  
- **Models**: Gemma‑3 (1 B, 4 B) and Qwen‑2.5 (0.5 B, 1.5 B, 3 B).  
- **Training**: LoRA rank 8, batch 1, 1,000 steps for FO, 100,000 steps for ZO.  
- **Device**: iPhone 15 Pro Max (8 GB DRAM).  
- **Metrics**: loss, next‑token accuracy, per‑step compute time, peak memory, decompression overhead, and impact of sequence length.

## Results

| Gemma3 1B | 33.01 % | 15.10 % | 11.92 % |

- **Utility**: FO converges 7× faster than ZO (loss/accuracy after 1,000 steps vs 100,000 steps).  
- **FFT**: 31.31 % loss, not better than LoRA.  
- **Performance** (Table 2):  
  - MeBP per‑step time is 43–94 % higher than MeZO, but MeBP requires 10×–100× fewer steps, yielding faster wall‑clock time.  
  - MeBP memory is up to 20 % higher than MeZO, yet all models stay below 1 GB.  
- **Decompression overhead** (Table 3): forward decompression 32–42 % of time, backward 13–24 %.  
- **Sequence length** (Table 4): compute time and memory grow with length; shorter sequences (≤ 512) are more suitable for on‑device fine‑tuning.

## Industry Impact  
Not reported.

## Limitations  
- Only single‑turn tasks were evaluated (ToolACE/BFCL live split).  
- Only LoRA fine‑tuning of all linear layers was considered; full‑parameter fine‑tuning was only a comparison point.  
- Experiments used 1,000 FO steps and 100,000 ZO steps; longer training regimes were not explored.  
- Sequence lengths beyond 512 were not investigated; efficient fine‑tuning on longer sequences remains future work.  
- More aggressive compression (e.g., 2‑bit) was not studied.  
- The approach is limited to the specific mobile device (iPhone 15 Pro Max) and Swift/iOS; portability to other platforms is untested.
