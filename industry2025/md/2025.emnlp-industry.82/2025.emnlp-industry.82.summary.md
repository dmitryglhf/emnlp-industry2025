# Summary

## TL;DR  
INSTAJUDGE is a Python library that aligns the judgment bias of LLM‑as‑Judge with human preferences for industrial use‑cases. It unifies two existing APO frameworks (DSPy and AdalFlow) and introduces a novel, cost‑efficient method called **Distribution‑Preserving Few‑Shot Sampling (DPFS)**. On four real‑world datasets, DPFS boosts human alignment by up to **64 %** over zero‑shot LLM‑as‑Judge and outperforms the DeepEval baseline by an average of **110 %**.

## Problem & Motivation  
Industrial AI systems often rely on LLM‑as‑Judge to replace human evaluation, but the judgment biases of LLMs are misaligned with those of humans. Fine‑tuning with human‑preference data is impractical in industry because it requires task‑specific, business‑oriented annotations and access to model logits, which closed APIs do not provide. Consequently, there is a need for a prompt‑level solution that can be applied to any LLM, including closed models, and that can be tuned with only a handful of human‑annotated examples.

## Approach  
- **Library Design**: INSTAJUDGE exposes a unified API that accepts a JSON prompt configuration (input fields, output fields, instruction) and an LLM client.  
- **APO Backbones**:  
  - **DSPy** – grid‑search + Bayesian optimisation over task instructions and few‑shot demos.  
  - **AdalFlow** – TextGrad‑style forward/backward pass that refines prompts via LLM‑generated feedback.  
- **DPFS** – a new APO method that:  
  1. **Samples** few‑shot examples from the human‑annotated set while preserving the label distribution (Algorithm 1).  
  2. **Selects** the best sample set by evaluating alignment on the training data (Algorithm 2).  
  3. Uses only **human‑written** examples, avoiding the need for LLM‑generated demos.  
  4. Reduces LLM calls to **O(n × K)** versus **O(n × N × I)** for DSPy/AdalFlow, yielding **2.2×** and **5.1×** fewer calls on average.  
- **Self‑Consistency Decoding** is supported to aggregate multiple reasoning paths for robust predictions.  
- **APO Options**: users can choose `engine` (`dspy`/`adal`), `instruction_opt`, `few_shot_opt` (`dspy`/`adal`/`dist_preserve`), and `eval_type` (`exact_match`/`fuzzy_match`).

## Experimental Setup  
| Dataset | # of data | Avg # of tokens |
|---------|------------|------------------|
| CCC | 1250 | 120 |
| CI  | 120  | 21  |
| GenAI | 111 | 220 |
| SearchQA | 100 | 6.1 K (1.5 K) |

- **LLM**: `gpt-4o-2024-05-13`.  
- **Evaluation Metrics**:  
  - Categorical dimensions → **F1‑score** (`F*w*`).  
  - Numerical dimensions → **Pearson correlation** (`P*r*`).  
- **Cross‑validation**: 5‑fold, reporting the mean of the five folds.  
- **APO Methods**:  
  - `Inst_only` (task instruction only).  
  - `Inst_demos` (instruction + demos).  
  - `DPFS` (only DPFS).  
  - `Inst_DPFS` (DPFS + instruction optimisation).  
- **Baselines**:  
  - Zero‑shot INSTAJUDGE with DSPy/AdalFlow engines.  
  - **DeepEval** (existing library, only numerical outputs).  
  - Opik omitted (uses G‑Eval, similar to DeepEval).  
- **Hyperparameters**: identical across methods; details in Appendix A.1.

## Results  
| Method | Avg. Gain vs. Zero‑shot | Avg. Gain vs. DeepEval |
|--------|--------------------------|------------------------|
| Inst_only | 27 % | – |
| Inst_demos | 36 % | – |
| DPFS | 56 % | – |
| Inst_DPFS | **64 %** | **110 %** |

- **DPFS** outperforms both DSPy and AdalFlow APO variants and requires **three times less running time** for APO.  
- **Inst_DPFS** consistently ranks top for each engine, closely followed by DPFS.  
- Zero‑shot INSTAJUDGE already surpasses DeepEval, highlighting the importance of bias alignment.  
- No significant improvement observed for the **RQ** dimension in the SearchQA use‑case.
