# Summary

## TL;DR  
RoSL is a plug‑and‑play framework that boosts SQL generation for small open‑source LLMs by inserting question decomposition into the schema‑linking stage. On the BIRD benchmark, RoSL raises execution accuracy from 37.31 % to 40.35 % (8.2 % absolute gain) and improves schema‑linking recall by 25.1 % using the ibm‑granite/granite‑3.3‑8b‑instruct model, making it a practical, privacy‑friendly solution for industry deployments.

## Problem & Motivation  
NL‑to‑SQL systems are vital for non‑technical users to query structured data, yet state‑of‑the‑art models (e.g., GPT‑4, Gemini) are proprietary, raising privacy concerns and demanding large GPU memory (up to 140 GB for 70B models). Smaller open‑source LLMs that fit on a single 80 GB GPU are more deployable, but existing work relies on fine‑tuning, limiting flexibility. Schema‑linking is often omitted for large models but remains essential for smaller ones; errors in this stage propagate to SQL generation. The paper addresses the need for a lightweight, fine‑tuning‑free method that improves schema‑linking for small LLMs while keeping deployment practical.

## Approach  
RoSL introduces a three‑agent pipeline:  
1. **Question Decomposer Agent** – decides whether a question needs decomposition and splits it into sub‑questions that can be joined to form the final query.  
2. **Schema Selector Agent** – for each sub‑question, selects a reduced schema containing only the relevant tables/columns.  
3. **SQL Generator Agent** – generates the final SQL using the union of all reduced schemas.  

Key design choices:  
- Decomposition is applied *before* schema‑linking (first in the literature).  
- The decomposition is constrained so that sub‑questions can be joined to reconstruct the original intent.  
- The framework is modular and uses instruction‑tuned open‑source LLMs, requiring no task‑specific fine‑tuning.

## Experimental Setup  
- **Datasets**: Spider and BIRD (full dev set and a subsampled dev split).  
- **Models**: Llama‑3.1‑8B, phi‑4‑14B, granite‑3.3‑8B, Qwen3‑4B, Llama‑3.1‑70B.  
- **Baselines**:  
  - **NoSL** – SQL generator receives the full schema without any linking.  
  - **SL** – standard schema linking using the selector from Talaei et al. (2024).  
  - **SL+QD (RoSL)** – our method with question decomposition at schema linking.  
  - Variants **SL+SimpleQD** (unconstrained decomposition) and **SL+TableQD** (single‑table sub‑questions).  
- **Metrics**: Execution Accuracy, Schema Linking Precision, Recall.  
- **Setup**: Modular prompting across stages; instruction‑tuned open‑source LLMs; evaluation on the subsampled_dev split for ablations.

## Results  
| Model | NoSL | SL | SL+QD (RoSL) |
|-------|------|----|---------------|
| Llama‑3.1‑8B | 46.08 | 47.19 | **49.28** |
| phi‑4‑14B | 51.36 | 59.77 | **59.91** |
| granite‑3.3‑8B | 37.28 | 37.31 | **40.35** |
| Qwen3‑4B | 44.98 | 49.28 | **49.93** |
| Llama‑3.1‑70B | 62.12 | 61.34 | 61.41 |

*Table 1 – Execution accuracy (%) on the BIRD dev set.*  
RoSL achieves the highest accuracy for three of the four open‑source models, with the largest absolute gain on granite‑3.3‑8B (3.07 % over SL).  

Schema‑linking recall (Table 2) shows that both unconstrained (Base QD) and structured (SL+QD) decompositions significantly boost recall over SL. For Llama‑3.1‑8B, recall rises from 0.7084 (SL) to 0.8309 (SL+QD).  

*Table 2 – Schema Linking Recall across decomposition variants.*  

The paper reports an overall 25.1 % recall improvement and 8.2 % execution accuracy improvement on BIRD using ibm‑granite/granite‑3.3‑8b‑instruct. No production or A/B‑test results are reported.
